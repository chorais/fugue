# **project-preamble.md — Project Preamble Template (Initiator Persona)**  
Version: 1.0  
Status: Draft  
Persona: Initiator  
Methodology: Fugue Orchestration Method  
Artefact Type: Project‑Level Preamble  

---

# **1. Mission**

Define the long‑term mission of the project.

Describe the enduring purpose, direction, and value the project intends to deliver.

State the overarching intent that will guide all Branches, Themes, and Tranches.

---

# **2. Vision**

Describe the desired future state the project aims to achieve.

Define the high‑level transformation, capability, or system outcome the project is intended to realise.

This section must be aspirational but bounded.

---

# **3. Scope**

## **3.1 In‑Scope**

List the domains, capabilities, behaviours, or concerns that the project will address.

These should be broad, multi‑theme areas of responsibility.

## **3.2 Out‑of‑Scope**

List the domains, capabilities, behaviours, or concerns that the project will explicitly exclude.

These exclusions must be clear enough to prevent downstream ambiguity.

---

# **4. Boundaries**

Define the structural, organisational, or conceptual boundaries of the project.

Clarify what the project is allowed to modify, influence, or govern — and what it must not.

Boundaries must be stable across the entire lifecycle.

---

# **5. Strategic Objectives**

List the long‑term objectives the project must achieve.

These should be durable, cross‑theme goals that remain valid across multiple tranches.

Each objective must be:

- measurable  
- durable  
- non‑overlapping  
- traceable  

---

# **6. Constraints**

List any known constraints that apply across the entire project.

These may include:

- regulatory constraints  
- architectural constraints  
- organisational constraints  
- technical constraints  
- operational constraints  

Constraints must be upstream signals for Project DECOR but must not contain DECOR content.

---

# **7. Risks & Considerations**

List any known risks, uncertainties, or considerations that may influence the project.

These may include:

- long‑term uncertainties  
- external dependencies  
- systemic risks  
- cross‑team risks  
- domain‑specific risks  

These are upstream signals for Project DECOR but must not contain DECOR content.

---

# **8. Dependencies**

List any upstream systems, organisational structures, external teams, or contextual requirements that the project depends on.

Dependencies must be structural and long‑term.

---

# **9. Success Criteria**

Define the measurable conditions under which the project will be considered successful.

Success criteria must be:

- objective  
- verifiable  
- durable  
- aligned with the Mission and Vision  

---

# **10. Notes for Downstream Personas**

Provide any clarifications, expectations, or contextual notes intended for:

- Architect (Project DECOR derivation)  
- Curator (governance envelope)  
- Branch creators  
- Theme creators  

This section must not contain architectural detail or governance enforcement.

---
